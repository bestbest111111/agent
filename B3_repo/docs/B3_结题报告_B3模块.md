### 2.2 系统整体运行流程与集成说明

B组Agent智能体系统运行流程：

1. 用户输入问题后，B1 Agent运行时启动消息循环
2. B1调B5获取相关记忆上下文
3. **B1调B3（本人负责）的`get_tools_schema()`生成tools_schema**（5个工具：calculator、file_reader、local_file_search、table_analyzer、format_converter）
4. B1将tools_schema和消息列表传给B4（Qwen3.5-4B），B4将schema绑定到模型
5. LLM返回AIMessage（含tool_calls或直接回答）
6. **若有tool_calls，B1调B3的`execute_tool_calls()`执行**：白名单校验→参数校验→动态导入B2 Skill→调用→返回ToolMessage
7. B1将ToolMessage追加到消息列表，再次调B4获取最终回答
8. 最终回答输出给用户，B1调B5保存对话记忆

在模块合并过程中，主要遇到了B3和B4之间tool_calls数据格式不一致的问题（OpenAI标准格式`{id, function: {name, arguments}}` vs 简化格式`{id, name, args}`），最终通过实现`normalize_tool_call()`统一处理解决。

## 三、 个人核心模块技术报告（个人成绩给定的核心依据）

### 3.1 模块定位与系统融合方式

*   **在系统中的角色**：

    我负责的**B3模块（说明生成与工具调用层）**是整个Agent系统的**桥接层**，位于B2（Skill工具函数层）和B4（LLM决策层）之间。该模块承担了不可替代的"翻译"和"调度"职责——将B2的普通Python函数翻译为LLM可识别的标准化JSON格式（tools_schema），再将LLM返回的工具调用指令校验后安全地派发给B2执行。没有本模块，LLM无法知道有什么工具可用，B2的5个Skill函数将变成孤立的脚本，整个Agent的"思考→调用工具→获取结果→继续思考"闭环无法形成。

*   **上下游依赖与接口协同**：

    | 方向 | 交互对象 | 数据格式 | 说明 |
    |:---:|:--------:|:--------:|:----|
    | 上游 | B4 LLM决策模块 | `tools_schema`（OpenAI Function Calling格式JSON数组） | B3生成，B1转发给B4绑定到模型 |
    | 上游 | B4 LLM（返回值） | `AIMessage.tool_calls` = `{id, name, args}` | B4返回，B1转发给B3执行 |
    | 下游 | B2 Skill工具函数 | `SkillResult` = `{skill_name, status, input, output, error, latency_ms}` | B3通过`invoke_callable()`调用B2 |
    | 协调 | B1 Agent运行时 | `ToolMessage` = `{role, tool_call_id, name, content, status}` | B3返回结果给B1 |

### 3.2 核心技术实现路径

*   **算法与工程实现**：

    本模块使用纯Python实现（510行核心代码 + 399行进阶代码），不依赖外部AI框架。核心技术如下：

    1. **配置文件管理**：使用PyYAML解析`tools.yaml`，校验tools和toolsets字段合法性，支持全局默认toolset和schema_source配置。

    2. **动态函数加载**：使用Python标准库`importlib.import_module()`根据YAML配置的`module`/`function`字段按字符串名动态导入B2的Skill函数。加新工具只需修改YAML和编写函数文件，B3代码无需变更，体现**开闭原则**（对扩展开放，对修改关闭）。

    3. **类型注解→JSON Schema转换（进阶功能）**：使用`inspect.signature()`读取函数签名，`get_type_hints()`读取类型注解，`get_origin()`/`get_args()`解析泛型类型（如`list[str]`、`str | int`、`Literal["a","b"]`），递归生成JSON Schema。支持两种模式：
       - `configured`模式：从YAML的`parameters`字段读取
       - `function`模式：从Python函数注解自动推导（本项目使用此模式）

    4. **三层参数校验引擎**：
       - 第一层：必填参数检查（`required`中的参数必须出现）
       - 第二层：未知参数检查（禁止传schema未定义的参数）
       - 第三层：类型/枚举/数值范围检查（`_schema_accepts()`递归校验）
       所有校验失败经`_result_from_exception()`统一封装为`SkillResult`错误格式，**不导致系统崩溃**。

    5. **Tool Call执行引擎**：每个tool_call独立处理，完整流程为：
       规范化格式 → 白名单校验 → 动态加载函数 → 生成参数schema → 参数校验 → 自动注入上下文参数 → 执行B2 Skill → 构造ToolMessage → 记录日志

    6. **上下文参数自动注入**：`data_root`（文件安全根目录）、`output_dir`（输出目录）、`limits_config`（资源限制）由B3自动从配置读取并注入到B2 Skill，LLM无需知道也不允许修改，保证文件操作安全。

    7. **去重缓存与统计（进阶功能）**：基于配置文件SHA256 + toolset + 工具名 + 参数生成缓存key，实现`success_only`缓存策略。输出统计报告包含success_rate、p50/p95延迟、error_code_counts。

    8. **Schema描述对比（进阶功能）**：生成完整描述和弱化描述两种schema变体，用真实Qwen3.5-4B模型测试两种方案的工具选择准确率。

*   **关键代码逻辑**：

    **核心一：动态加载B2 Skill函数**
    ```python
    def _load_callable(definition):
        """根据tools.yaml配置动态导入B2函数"""
        module_name = definition.get("module")      # "skills.calculator"
        function_name = definition.get("function")  # "calculator"
        module = importlib.import_module(module_name)
        return getattr(module, function_name)
    ```
    *原理：根据tools.yaml配置的module/function字段，用`importlib.import_module()`在运行时按字符串名动态导入B2模块并获取函数对象。加新工具只需改YAML，B3代码不动。*

    **核心二：从函数注解自动生成参数Schema**
    ```python
    def _function_parameter_schema(function, definition):
        """从Python函数签名自动生成参数schema"""
        signature = inspect.signature(function)   # 读函数签名
        hints = get_type_hints(function)           # 读类型注解
        for name, parameter in signature.parameters.items():
            if name in INJECTED_PARAMETERS:  # 跳过自动注入参数
                continue
            annotation = hints.get(name)
            generated = _annotation_schema(annotation)  # str→{"type":"string"}
            if parameter.default is inspect.Parameter.empty:
                required.append(name)  # 无默认值→必填
            else:
                property_schema["default"] = parameter.default  # 有默认值
    ```
    *原理：用`inspect.signature()`读取函数签名（如`calculator(expression: str) -> dict`），`get_type_hints()`读取类型注解，自动推导JSON Schema：无默认值参数加入required，有默认值记录default值。*

    **核心三：三层参数校验**
    ```python
    def _validate_args(args, parameter_schema):
        properties = parameter_schema.get("properties", {})
        required = parameter_schema.get("required", [])
        # 第一层：必填参数检查
        missing = [n for n in required if n not in args]
        # 第二层：未知参数检查
        unknown = sorted(set(args) - set(properties))
        # 第三层：类型/枚举/范围检查
        for name, value in args.items():
            if not _schema_accepts(properties[name], value):
                raise SkillFault(...)
    ```
    *原理：三层安全拦截——必填缺失返回具体提示、未知参数拒绝、类型不匹配报错。所有异常封装为统一SkillResult格式。*

*   **进阶挑战攻克**：

    1. **自动从Python函数注解生成schema**：使用`inspect.signature()`+`get_type_hints()`读取函数签名，`get_origin()`+`get_args()`递归解析`list[str]`、`str|int`、`Literal`等复杂泛型类型，实现了从源码直接推导参数类型，无需手动在YAML中编写parameters。实现了configured（从YAML读取）和function（从注解推导）两种模式的自由切换。

    2. **tool_call去重缓存与统计**：基于配置文件SHA256+toolset+name+args的JSON序列化结果生成SHA256缓存key，仅对calculator等确定性工具开启缓存（`cacheable:true`），实现`success_only`缓存策略（第1次miss后执行并写入，第2次命中缓存直接返回）。同时统计success_rate、p50/p95延迟、error_code_counts等指标。

    3. **Schema描述对比**：生成完整描述和仅名称描述两种schema变体，通过真实Qwen3.5-4B模型测试两种方案的工具选择准确率。验收结果：full_description准确率100%，name_only准确率100%。

## 四、 实训总结与心得体会

### 4.1 个人实训收获与挑战

*   **遇到的最大挑战**：

    在模块开发中，最大的挑战是B3和B4之间tool_calls数据格式的统一问题。OpenAI标准格式为`{id, function: {name, arguments}}`（arguments是字符串），而项目中使用的简化格式为`{id, name, args}`（args是字典）。同时，B2 Skill函数的参数类型需要精确匹配JSON Schema定义，否则LLM传的参数可能在校验阶段被拒绝。

*   **如何克服的**：

    通过实现`normalize_tool_call()`函数统一处理两种输入格式，并在`_function_parameter_schema()`中从函数注解自动推导参数类型，确保schema与B2函数签名严格一致。同时，`_schema_accepts()`函数对参数值进行类型自动转换（如int转float），提高了容错性。在接口规范上，与B1和B4的同学提前约定了ToolMessage和AIMessage的消息契约格式，减少联调问题。

*   **心得体会**：

    这次实训让我深入理解了Agent系统中"工具调用"这一核心环节的实现原理。B3作为桥接层，需要兼顾LLM的灵活性和工程的安全性，使用`importlib`动态加载、`inspect`反射读取函数签名等Python元编程技术，让我对动态语言的灵活性有了更深的认识。团队协作中，模块间接口的统一定义和格式规范至关重要，提前约定数据格式可以避免后期大量联调问题。

    本次实训我实现了进阶功能的全部三个挑战项（自动生成schema、缓存统计、描述对比），验收结果为`status: success`，全部通过。
